Stella Frutta - Specifica Database

Versione: 1.0

Data: 01/04/2026

Scopo: definire il modello dati relazionale del Core Stella Frutta come base per database SQL Server, Entity Framework, migrazioni, repository e API.

1. Obiettivi del modello dati

  • Supportare in modo coerente prodotti, lotti, genealogia, magazzino, qualità, ordini e audit.
  • Garantire tracciabilità completa dei lotti a monte e a valle.
  • Supportare segregazione BIO/non BIO.
  • Consentire evoluzione graduale del dominio senza refactor distruttivi.
  • Rendere semplice la traduzione in Entity Framework e SQL Server.

2. Convenzioni generali

  • Tutte le tabelle principali hanno chiave primaria Id di tipo intero o bigint identity.
  • Le date tecniche usano datetime2.
  • I campi monetari o quantitativi usano decimal.
  • I campi stato e tipo sono preferibilmente nvarchar con vincoli applicativi oppure enum mappati a stringa.
  • I codici pubblici come Sku e BatchCode devono avere vincolo di univocità.
  • Le cancellazioni fisiche vanno evitate sulle entità core; preferire stato o archiviazione logica.

4. Schema concettuale sintetico

Relazioni principali

  • Un Customer ha molti CustomerAddresses.
  • Un Product ha molti Batch.
  • Un Supplier ha molte Certification e puo fornire molti Batch.
  • Un Batch puo avere molti BatchLink in ingresso e in uscita.
  • Un Batch ha molti InventoryMovement, QualityCheck e NonConformity.
  • Un Customer ha molti Order.
  • Un Order ha molti OrderItem.
  • Un Order e una OrderItem possono avere una o più StockReservation.
  • Le entità critiche producono record in AuditLog.

10. Vincoli funzionali principali

  1. Products.Sku deve essere univoco.
  2. Batches.BatchCode deve essere univoco.
  3. BatchLinks non puo collegare un lotto a se stesso.
  4. Ogni InventoryMovement deve avere prodotto, lotto, tipo, quantita, data e autore.
  5. Una NonConformity chiusa deve avere almeno ClosedAt o ClosedBy.
  6. Le certificazioni BIO valide devono essere verificabili rispetto alla data di utilizzo del lotto.
  7. Può esistere un solo indirizzo IsDefault = 1 per CustomerId + AddressType.
  8. Una StockReservation attiva non può superare la disponibilità residua del prodotto/lotto.
  9. Una riserva Consumed o Released non può tornare attiva senza un nuovo record o una riapertura esplicita.

12. Vincoli e scelte EF / implementazione

  • Mappare gli enum applicativi a stringa leggibile in database dove utile.
  • Configurare esplicitamente le relazioni su BatchLink con doppia FK verso Batches.
  • Configurare Customer -> CustomerAddresses come 1:N con delete behavior restrittivo o controllato.
  • Configurare Order/OrderItem -> StockReservations come 1:N senza cascade delete distruttiva.
  • Evitare cascade delete aggressive sulle entità core.
  • Usare decimal(18,3) per quantita operative.
  • Usare decimal(18,2) per importi.
  • Valutare rowversion su tabelle sensibili se si vuole gestire concorrenza ottimistica.

13. Query critiche da supportare

  • Ricerca lotto per codice.
  • Ricostruzione genealogia a monte e a valle.
  • Calcolo disponibilita prodotto/lotto.
  • Calcolo disponibilità vendibile = stock reale - riserve attive.
  • Lista ordini per stato.
  • Lista non conformità aperte.
  • Storico audit per entità.

14. Decisioni aperte sul database

  • gestione multi-magazzino o singolo deposito logico
  • gestione indirizzi cliente in tabelle dedicate o embedded
  • modello checklist qualità dettagliato o semplificato nella prima fase
  • eventuale tabella dedicata per riserve stock invece di solo movement type

15. Output attesi dal prossimo step

  1. Traduzione del modello in classi entity e configurazioni EF.
  2. Creazione delle prime migration SQL Server.
  3. Definizione degli endpoint sulle tabelle core.
  4. Scelta di eventuali tabelle ausiliarie mancanti.
